# Confluence export

Confluence Cloud storage-format pages for the design corpus, plus a converter and a bulk push/update script.

## Layout

```
confluence/
├── README.md          this file
├── convert.py         Markdown → Confluence storage format
├── push.py            bulk create/update against the Confluence Cloud REST API
├── manifest.json      filename → page title (written by convert.py)
├── state.json         filename → {id, version} (written by push.py)
└── pages/             16 storage-format .xml files (regenerated by convert.py)
```

## Workflow

1. **Convert** — produces or refreshes the storage-format files:
   ```sh
   python3 convert.py
   ```

2. **Push** — creates pages on first run, updates them on subsequent runs:
   ```sh
   export CONF_URL=https://yourorg.atlassian.net
   export CONF_EMAIL=you@example.com
   export CONF_TOKEN=<atlassian-api-token>
   export SPACE_KEY=DESIGN
   # optional: a parent page id to place the corpus under an existing page
   # export PARENT_ID=123456

   # safe first preview
   DRY_RUN=1 python3 push.py

   # for real
   python3 push.py
   ```

3. **Edit and update** — modify the Markdown in `../content/`, then:
   ```sh
   python3 convert.py && python3 push.py
   ```
   `push.py` reads `state.json` and issues `PUT` updates rather than re-creating pages.

## Auth

Create an API token at <https://id.atlassian.com/manage-profile/security/api-tokens>. The script uses HTTP Basic auth with `email:token`.

The token needs write access to the target space. For most users that means a personal token; for service-account installs the same flow applies.

## What converts to what

| Markdown                              | Confluence                                                       |
|---------------------------------------|------------------------------------------------------------------|
| First H1 of the file                  | The page title (set on create/update)                            |
| Other ATX headings                    | `<hN>` (markdown H2 → Confluence h1, demoted by one level)       |
| Paragraphs, lists, tables             | `<p>`, `<ul>`/`<ol>`, `<table>`                                  |
| Bold, italic, inline code             | `<strong>`, `<em>`, `<code>`                                     |
| Fenced code blocks (any language)     | `code` macro with `language` parameter                           |
| Fenced ```mermaid blocks              | `mermaid-cloud` macro (Stiltsoft Mermaid Diagrams for Confluence)|
| Blockquotes starting with `*In plain terms:*` etc. | `info` panel macro                                  |
| Other blockquotes                     | `<blockquote>`                                                   |
| Links to `OTHER.md`                   | Confluence page link by page title                               |
| External links                        | `<a href="...">...</a>`                                          |
| Horizontal rules                      | `<hr/>`                                                          |

## Configuring the Mermaid macro

The default mermaid macro name is `mermaid-cloud`, which matches the Stiltsoft *Mermaid Diagrams for Confluence (Cloud)* app. If your Confluence Cloud instance uses a different mermaid app — for example one that registers the macro as `mermaid` — change the constant at the top of `convert.py`:

```python
MERMAID_MACRO = "mermaid-cloud"  # or "mermaid", etc.
```

Then re-run `convert.py`.

## Page hierarchy

The 16 pages are created with `index.xml` (the corpus index) as the parent and pages `01_` through `15_` as its children, in numeric order. If you set `PARENT_ID` to an existing Confluence page, that becomes the grandparent — the index sits beneath it and is itself the parent of the rest.

## Idempotence

- `state.json` is the source of truth for "which Confluence pages correspond to which local files."
- If a page is renamed in Confluence (its title changes), `push.py` will update it correctly because it identifies pages by ID, not title. The next push will also rename it back to the manifest title.
- If `state.json` is deleted, the next run will try to *create* pages with the manifest titles. Confluence requires titles to be unique within a space, so this will fail until the existing pages are removed or renamed.
- To force a clean re-creation: delete all the existing pages in Confluence (or move them to a different parent), then delete `state.json` locally.

## Limitations

- Anchors (`#section`) within cross-page links are dropped — Confluence anchors work differently and the corpus only uses anchors for in-page scrolling, not as page targets.
- The converter handles the Markdown subset used in this corpus. If you add features (nested lists, footnotes, definition lists, HTML embeds), the converter may need extending.
- Confluence Cloud requires unique page titles within a space; the converter uses each file's first H1 as the title. If two files share a title, the second will fail to create. Run `convert.py` and check `manifest.json` to see all titles before pushing.
