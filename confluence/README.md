# Confluence export

Confluence Cloud storage-format pages for the design documentation, plus a converter and a bulk push/update script.

## Layout

```
confluence/
├── README.md                            this file
├── convert.py                           Markdown → Confluence storage format
├── push.py                              bulk create/update against the Confluence Cloud REST API
├── push-all.sh                          wrapper: pre-flight + convert + push to every configured space
├── manifest.json                        filename → {title, attachments} (written by convert.py)
├── state-<host>-<space>.json            filename → {id, version, attachments} per target space (written by push.py, gitignored)
├── diagrams/                            rendered PNGs + .mmd sources (gitignored, regenerated by convert.py)
└── pages/                               19 storage-format .xml files (regenerated by convert.py)
```

## Workflow

1. **Convert** — produces or refreshes the storage-format files:
   ```sh
   python3 convert.py
   ```

2. **Push** — creates pages on first run, updates them on subsequent runs (see [Credentials](#credentials) below for the env vars):
   ```sh
   # safe first preview
   DRY_RUN=1 python3 push.py

   # for real
   python3 push.py
   ```

3. **Edit and update** — modify the Markdown in `../content/` (or `../private/intro-letter.md` for the Confluence-only intro), then:
   ```sh
   python3 convert.py && python3 push.py
   ```
   `push.py` reads the per-space state file and issues `PUT` updates rather than re-creating pages.

## Credentials

API tokens do NOT belong in this repository, even in the gitignored `private/` folder. push.py reads its configuration from environment variables; the recommended pattern is to keep per-space `.env` files in your XDG config directory.

### Layout

```
~/.config/atlassian/                   # chmod 700
├── planningvic-cng.env                # Planning Victoria, CNG space        (chmod 600)
└── publictransportvic-cng.env         # Public Transport Victoria, CNG space (chmod 600)
```

Each file holds the four variables push.py expects:

```sh
# example: planningvic-cng.env
CONF_URL=https://planningvic.atlassian.net
CONF_EMAIL=you@yourorg.com
CONF_TOKEN=ATATT3xFfGF0...
SPACE_KEY=CNG
# PARENT_ID=12345    # optional: places the design under an existing page rather than at the space root
```

Get a Confluence Cloud API token at <https://id.atlassian.com/manage-profile/security/api-tokens>. The script uses HTTP Basic auth with `email:token`. The token needs write access to the target space.

### Pushing to one space

```sh
set -a; source ~/.config/atlassian/planningvic-cng.env; set +a
cd ~/Development/cloud-native-geospatial-design/confluence
python3 convert.py            # regenerate pages/*.xml
DRY_RUN=1 python3 push.py     # safe preview — prints what would happen
python3 push.py               # real run
```

The `set -a; source ...; set +a` pair tells the shell "export every variable assigned in the sourced file to subprocesses." Without it, the variables would only exist for the current shell line, not for `python3`.

### Pushing to multiple spaces

The two target spaces for the current handover:

| Space | URL |
|---|---|
| Planning Victoria — CNG | <https://planningvic.atlassian.net/wiki/spaces/CNG/overview> |
| Public Transport Victoria — CNG | <https://publictransportvic.atlassian.net/wiki/spaces/CNG/overview> |

The recommended path is the wrapper script `push-all.sh`:

```sh
cd ~/Development/cloud-native-geospatial-design/confluence
./push-all.sh              # dry run — shows what would happen for both spaces
./push-all.sh --apply      # real push to every configured space
```

It runs the pre-flight check (env files present, CONF_EMAIL placeholder replaced, CONF_TOKEN non-empty), regenerates the pages once, then pushes to each space inside its own subshell so env vars from one cannot leak into the next. Targets are listed at the top of the script — add more spaces by appending to the `TARGETS` array.

If you prefer to do it manually:

```sh
# Planning Victoria
set -a; source ~/.config/atlassian/planningvic-cng.env; set +a
python3 push.py

# Public Transport Victoria
set -a; source ~/.config/atlassian/publictransportvic-cng.env; set +a
python3 push.py
```

push.py writes its page-id mapping to a **per-(host, space) state file** — `state-planningvic_atlassian_net-CNG.json` and `state-publictransportvic_atlassian_net-CNG.json` respectively. Each space's state is independent; the same local edits can be re-pushed to both at any time without their page IDs interfering. Both state files are gitignored.

## What converts to what

| Markdown                              | Confluence                                                       |
|---------------------------------------|------------------------------------------------------------------|
| First H1 of the file                  | The page title (set on create/update)                            |
| Other ATX headings                    | `<hN>` (markdown H2 → Confluence h1, demoted by one level)       |
| Paragraphs, lists, tables             | `<p>`, `<ul>`/`<ol>`, `<table>`                                  |
| Bold, italic, inline code             | `<strong>`, `<em>`, `<code>`                                     |
| Fenced code blocks (any language)     | `code` macro with `language` parameter                           |
| Fenced ```mermaid blocks              | `mermaid-cloud` macro (Stiltsoft Mermaid Diagrams for Confluence)|
| Blockquotes starting with `*In plain terms:*` etc. | `info` panel macro                                  |
| Other blockquotes                     | `<blockquote>`                                                   |
| Links to `OTHER.md`                   | Confluence page link by page title                               |
| External links                        | `<a href="...">...</a>`                                          |
| Horizontal rules                      | `<hr/>`                                                          |

## Public vs Confluence intro

`content/index.md` (the public Quartz homepage) is a sanitised version of the introduction with department references, the specific dollar figure, and the departure note removed.

`private/intro-letter.md` is the personal version intended for the Confluence space — same structure, retains all the context.

`convert.py` reads from `content/` by default. A `SOURCE_OVERRIDES` mapping at the top of `convert.py` swaps in `private/intro-letter.md` as the source for `index.xml`. This means a single `python3 convert.py` invocation always produces the Confluence-appropriate intro; the Quartz site is independent and never sees the personal version.

## Mermaid diagrams: pre-rendered to images

Confluence Cloud does not render mermaid natively, and the destination spaces for this handover cannot install marketplace apps. To remove the dependency entirely, `convert.py` pre-renders every fenced ```` ```mermaid ```` block to a PNG via **mermaid-cli** (`mmdc`), and emits an `<ac:image><ri:attachment.../></ac:image>` reference in the storage-format XML. `push.py` uploads each PNG as a page attachment after creating/updating the page, so the image renders inline regardless of which apps the Confluence space has installed.

### Dependency

```sh
npm install -g @mermaid-js/mermaid-cli
```

mermaid-cli ships its own headless Chromium via puppeteer; first install is large (~350 packages) but subsequent runs are fast. `convert.py` checks `mmdc` is in `PATH` and exits with a clear install message if not.

### How it works

1. `convert.py` walks the markdown, extracts each mermaid block, names it deterministically (`{page}-mermaid-{N}.png`), writes the source to `diagrams/{page}-mermaid-{N}.mmd`, and renders it to `diagrams/{page}-mermaid-{N}.png` via `mmdc -w 1600 -b white`.
2. In the storage-format XML, the mermaid macro is replaced with `<ac:image ac:align="center"><ri:attachment ri:filename="..."/></ac:image>`.
3. `manifest.json` records each page's `attachments` list.
4. `push.py` reads the attachment list per page. After creating or updating the page, it uploads any new PNGs (multipart POST), updates PNGs whose SHA-256 has changed since the previous push, and deletes orphans (attachments referenced by an earlier push but no longer in the manifest).
5. The state file tracks `{filename: {id, hash}}` per attachment so unchanged diagrams are skipped on re-runs.

### Resolution

Default width is 1600 px; height auto-scales. Change `DIAGRAM_WIDTH` in `convert.py` if you need larger or smaller. SVG output is supported by `mmdc` if you prefer — change the file extension and the storage-format mime type in `_parse_fence`.

## Page hierarchy

`index.xml` (the introduction) becomes the parent. `00_index.xml` (the document index) and pages `01_` through `18_` are created as its children, in `PAGE_ORDER` from push.py. If you set `PARENT_ID` to an existing Confluence page, that becomes the grandparent — the introduction sits beneath it and is itself the parent of the rest.

## Idempotence

- The per-(host, space) state file is the source of truth for "which Confluence pages correspond to which local files in this space."
- If a page is renamed in Confluence (its title changes), `push.py` will update it correctly because it identifies pages by ID, not title. The next push will rename it back to the manifest title.
- If a state file is deleted, the next run will try to *create* pages with the manifest titles. Confluence requires titles to be unique within a space, so this will fail until the existing pages are removed or renamed.
- To force a clean re-creation: delete all the existing pages in Confluence (or move them to a different parent), then delete the relevant state file locally.

## Limitations

- Anchors (`#section`) within cross-page links are dropped — Confluence anchors work differently and the documentation only uses anchors for in-page scrolling, not as page targets.
- The converter handles the Markdown subset used in this design. If you add features (nested lists, footnotes, definition lists, HTML embeds), the converter may need extending.
- Confluence Cloud requires unique page titles within a space; the converter uses each file's first H1 as the title. If two files share a title, the second will fail to create. Run `convert.py` and check `manifest.json` to see all titles before pushing.
